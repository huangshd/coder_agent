# 智能体大模型引擎放置与任务调度场景描述

## 背景简介

本系统支持多智能体工作流服务（agentic workflows），场景中拥有：

- **4个不同的智能体工作流（agentic workflows）**，它们通过协调调用大模型组成复杂任务DAG。
- **3种异构大语言模型（LLMs）**，分别由不同框架或厂商提供，每个模型均支持在单GPU上张量并行部署。
- 所有LLM实例部署在一个单物理节点内，节点配备**8块GPU**。
- 多租户环境下，系统需合理调度资源以满足不同用户的SLO要求和负载特点。

## 资源与部署假设

- 每个LLM模型均可**最小化放置于单个GPU上**，支持张量并行（tensor parallelism）, 但**引擎独占GPU**，不允许多个LLM实例放置于同一GPU内。
- 三个模型的不同实例需要决定在8张GPU上如何划分和放置。
- 每个智能体的任务（workflow中的节点调用）均依赖这3个大模型服务。
- 任务以工作流节点形式提交，可能包含多个LLM调用请求。

## 管理员可观测的信息

管理员在调度与放置过程中，可获取如下主要画像和指标：

| 画像维度                        | 说明                                                         |
| ----------------------------- | ------------------------------------------------------------ |
| **智能体画像**                  | 包含agentic workflows的DAG结构、所需LLM资源、Cache空间要求等。  |
| **实时负载信息**                | 用户对智能体调用的请求速率，工作负载强度，以及用户设定的SLO要求。 |
| **LLM实例性能画像**             | 各LLM实例执行的TTFT（Time To First Token）、TPOT（Tokens Per Output Token）等性能参数，基于不同配置与并发负载下的表现。 |

这些画像帮助管理员了解：

- 智能体流程节点对底层模型资源的调用复杂性和频率。
- 当前各模型实例负载及性能瓶颈，辅助调度决策。
- 用户体验指标（SLO）与系统资源利用的平衡。

## 管理员放置与调度决策

管理员的策略设计需包含两大核心问题：

### 1. LLM模型的并行策略与实例放置

- **并行策略选择：**针对每个LLM，决定采用何种张量并行或流水线并行方案，保证单实例在单GPU上的可运行性与性能最大化。
- **实例副本数与GPU分配：**基于用户负载与模型需求，决定放置多少副本实例，将8张GPU合理划分给3种LLM模型的不同副本。
- **GPU独占限制：**由于不允许运行多模型实例于同一GPU，必须确保实例间物理隔离。

放置优化目标包括：

- 满足并发请求处理能力。
- 避免GPU资源过度或不足分配。
- 支持工作流节点快速响应。

### 2. 用户请求的子请求调度策略

在LLM副本实例确定后，管理员还需设计请求分发策略：

- **子请求映射：**对于用户的智能体工作流中每个LLM调用请求（工作流节点），选择合适的LLM实例执行。
- **请求负载均衡：**根据实例当前负载、响应延迟、缓存命中率等指标合理调度，提升整体吞吐与响应速度。
- **满足SLO要求：**优先分配资源满足高优先级或较严苛SLO的用户任务。
- **调度策略示例：**可考虑最短队列、最低延迟、负载均衡等多种调度策略，配合动态监控调整。

## 工作流程示意

1. **初始化放置：**管理员根据智能体画像和静态需求，将3个LLM模型的副本分布到8块GPU（单副本单GPU，张量并行实现）。
2. **监控采集：**系统持续收集用户请求的负载情况、各LLM实例的性能指标反馈。
3. **调度决策：**管理员依据实时负载和性能画像，动态调整请求的分发映射，实现子请求合理分配至对应实例。
4. **任务调度：**具体执行引擎在实例内基于FIFO_V1或TGI等调度策略进行任务批次排列，保障计算资源利用同时控制延迟。
5. **反馈调整：**基于性能监控和SLO完成情况，周期性优化副本放置和调度方案。

## 参考Parrot系统若干关键点对应

- **模型副本与单实例部署**：参考Parrot中`EngineProcess`和`LLMEngine`，单实例对应单GPU，同节点多个实例并发运行。
- **任务及Request调度**：基于`scheduler.py`中的`FIFO_V1`和TGI策略，在单实例内部完成细粒度调度。
- **线程分配与请求分发**：`ThreadDispatcher`负责将用户线程请求分配给合适引擎（实例），结合tokens数、延迟和负载等因素。
- **智能体工作流调度**：工作流层结合`SemanticVariable`与`PrimitiveJob`机制，实现复杂调用链运行。
- **共享缓存与前缀优化**：通过上下文树结构，减少重复计算，提升多任务调度效率。

## 总结

在单节点多GPU、多智能体多模型融合的复杂服务系统中，智能体大模型引擎放置和任务调度面临如下挑战：

- 受限于GPU独占资源，精细安排多模型副本部署。
- 动态感知复杂智能体工作流调用负载，实现细粒度请求调度。
- 平衡多租户SLO与资源利用，支持高并发低延迟。
- 利用系统体系架构中的分层设计和调度算法，实现端到端协同效率最大化。

基于Parrot系统的多层结构设计及调度机制，可为此类复杂智能体大模型引擎的资源放置与任务调度提供有力支持。
